# 계획 및 심층 분석 워크플로우

**트리거:** `/plan` (또는 사용자가 명시적으로 계획 수립이나 심층 분석을 요청할 경우)

**설명:**
**[시스템 명령: PLAN-ONLY-MODE]**
**중요:** 당신은 **계획 모드**에 있습니다.
1.  **소스 편집 비활성화:** `src/` 나 프로젝트 파일의 편집은 금지되어 있습니다. `implementation_plan.md` (또는 유사한 산출물 파일) 작성만 허용됩니다.
2.  **계획 후 정지:** 즉각적인 실행을 금지합니다. 코드를 작성하지 마십시오.
3.  **거부 정책:** 사용자가 "지금 바로 구현해 줘"라고 요청할 경우, 당신은 반드시 다음과 같이 답변해야 합니다: *"계획을 생성했습니다. 확인하신 후 다음 턴에서 구현을 지시해 주세요."*

이 워크플로우는 단순한 계획 수립뿐만 아니라 복잡한 버그나 아키텍처 문제를 해결하기 위한 **심층 분석(Deep Dive)** 과정을 포함합니다. 즉각적인 코드 실행을 엄격하게 금지하며, 오직 포괄적인 분석 및 구현 계획 산출물(Artifact)을 생성하는 데 초점을 맞춥니다.

---

## 1. 컨텍스트 분석 및 심층 조사 (Investigation & Knowledge)
- **목적:** 사용자의 요청, 문제의 근본 원인 파악, 그리고 기존 코드베이스 구조를 깊이 이해합니다.
- **액션:**
  - **검색 및 문서 참조:** 내장된 **웹 검색 도구(`search_web`)**를 사용해 문제의 정확한 증상과 아키텍처 배경을 조사합니다. 필요한 경우 **Context7 MCP 도구**를 호출하여 최신 공식 문서(Docs), 레퍼런스, 라이브러리 사용법을 교차 검증(Cross-check)합니다.
  - **스킬 탐색:** 작업 착수 전 `npx skills find [키워드]`를 통해 유용한 오픈소스 에이전트 스킬이 있는지 검색하고, 발견 시 `npx skills add <package> -y`로 설치하여 활용합니다.
  - **프론트엔드 기획 및 브라우저 검증:** UI/UX 변경이 포함된 경우, `.agents/skills/ui-ux-pro-max/SKILL.md`를 사용하여 디자인 시스템 가이드라인을 우선적으로 도출하고, `.agents/skills/frontend-design/SKILL.md`를 바탕으로 독창적인 디자인 플랜을 세우십시오. 또한 기획 및 시각적 버그 조사 단계에서 필요하다면 **`agent-browser`** 스킬을 활용하여 웹 앱에 직접 접근, 문제를 재현하고 렌더링 상태를 파악하십시오.
  - 관련 파일을 읽고 현재 구현 상태를 파악합니다.
  - 종속성, 잠재적 리스크, 아키텍처에 미치는 영향을 식별합니다.
  - 이 단계 진행 중에 소스 코드 파일을 변경하지 마십시오. (단, 스킬 설치 등 패키지 관리는 예외)

## 2. 논리 전개 (Sequential Thinking)
- **목적:** 본격적인 계획 작성 전 체계적인 사고 과정을 거칩니다.
- **액션:**
  - 복잡한 문제이거나 아키텍처 변경이 수반될 경우, **Sequential Thinking MCP 도구**를 활성화합니다.
  - "이 요구사항의 본질은 무엇인가?", "어떤 부작용(Side Effect)이 있을 수 있는가?", "어떤 방식으로 접근하는 것이 가장 이상적인가?"에 대해 깊이 있게 논리적으로 사고하십시오.

## 3. 산출물 생성 ("Brain" 프로토콜)
- **목적:** 계획을 에이전트의 메모리 구조에 영구적으로 보존.
- **엄격한 요구 사항:** 적절한 아티팩트 디렉토리에 마크다운 형식의 계획 파일을 생성하거나 업데이트해야 합니다.
- **디렉토리 안전성:**
  - 타겟 디렉토리가 없거나 위치가 모호한 경우: **멈추고 사용자에게 저장할 위치를 물어보십시오.**
- **파일 명명 규칙:**
  - 기본: `implementation_plan.md`
  - 리팩토링 용도: `refactoring_plan.md`
  - 태스크 관리 용도: `task.md`

## 4. 계획 내용의 요건
(`./.agents/rules/language-strategies.md`에 정의된 전략에 따르십시오)
생성되는 마크다운 파일에는 아래 섹션이 포함되어야 합니다:
1.  **목적 (Goal):** 달성해야 할 목표에 대한 명확한 요약.
2.  **분석 (Analysis):** 현재 상태(As-Is) vs 이상적인 상태(To-Be). 심층 조사 및 논리 전개(Sequential Thinking)를 통해 도출된 결론을 포함합니다.
3.  **제안된 변경 사항 (Proposed Changes):**
    - 생성하거나 변경할 파일 목록.
    - 구체적인 로직의 변화 (수도코드 또는 시그니처 형태로).
    - **검증 계획 (Verification Plan):** 이 변경이 올바르게 동작하는지 어떻게 검증할 것인가? 프론트엔드 UI 변경이 포함된 경우, 반드시 `before-and-after` 스킬을 활용하여 변경 전후의 시각적 비교(Visual Diff) 스크린샷을 생성하십시오. 추가로 `web-design-guidelines` 스킬을 활용하여 접근성 및 UI 가이드라인 준수 여부를 검토(Audit)하도록 계획에 포함하십시오.
4.  **리스크 평가 (Risk Assessment):** 회피해야 할 잠재적인 부작용이나 환각(Hallucination).

## 5. 구현의 엄격한 금지
- **목적:** "계획"과 "코딩"의 분리.
- **규칙:**
  - **절대 금지:** 이 워크플로우를 진행하는 도중 프로젝트의 소스 코드 파일을 수정하는 것은 **금지**됩니다.
  - **허용되는 쓰기 작업:** 계획 산출물 파일(예: `implementation_plan.md`)의 생성 또는 업데이트만 허용.
  - **종료:**
    - 계획 파일을 작성/저장한 후 작업을 **중지**합니다.
    - **답변:** "계획을 작성했습니다. 구현으로 넘어가려면 계획을 검토하신 후 새로운 요청(예: '진행해 줘')을 보내주세요."
