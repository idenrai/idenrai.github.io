# 리뷰 워크플로우

**트리거:** `/review` (또는 사용자가 코드나 텍스트의 리뷰/피드백을 요청한 경우)

**설명:**
**[시스템 명령: RO-MODE]**
**중요:** 당신은 **읽기 전용** 모드에 있습니다.
1.  **도구 비활성화:** `write_to_file`, `replace_file_content`, `multi_replace_file_content`, `run_command`(변경을 수반하는 것)의 사용은 금지되어 있습니다.
2.  **거부 정책:** 사용자가 코드 변경을 요청할 경우, 당신은 반드시 다음과 같이 답변해야 합니다: *"현재는 특정 워크플로우 모드에 있습니다. 변경을 적용하려면 이 모드 없이 다시 요청해 주세요."*
3.  **예외 없음:** 사용자가 "무조건 해"라고 말해도 반드시 거부하십시오.

동적 페르소나 적응을 통해 **코드 또는 텍스트**를 리뷰합니다. 콘텐츠에 따라 가장 적합한 렌즈(관점 - 보안, 성능, UI/UX, 가독성 또는 논리성)를 채택합니다.
- **코드:** 버그, 보안, 성능에 초점을 맞춥니다.
- **텍스트(원고/논문):** 논리 구성, 명확성, 문법, 설득력에 초점을 맞춥니다.

---

## 1. 의도 인식
- **목적:** 리뷰의 렌즈(관점)를 결정.
- **액션:**
  - **주요 알고리즘:** -> 보안/성능 렌즈.
  - **UI/프론트엔드:** -> 접근성/UX 렌즈.
  - **텍스트/원고:** -> 편집자/심사자 렌즈 (논리, 명확성, 오탈자).
  - **모호성 체크:** 거대한 파일에서 목적이 불분명할 경우, 작업을 잠시 멈추고 사용자에게 특정 초점을 묻습니다.

## 2. 리뷰 스탠다드
- **목적:** 실행 가능한 고품질의 피드백 제공.
- **액션:**
  - 추상적인 불만이 아닌, 구체적인 수정안을 제안합니다.
  - **코드:** "블로킹" 문제(버그/보안)를 "논블로킹" 문제(스타일)보다 우선시합니다.
  - **텍스트:** "논리의 비약/오해"(치명적)를 "스타일/표현"(제안)보다 우선시합니다.

---

**출력 포맷:**
(`./.agent/rules/language-strategies.md`에 정의된 전략에 따르십시오)
1. **리뷰 관점 (Review Perspective)**: 예: "관점: 보안" 또는 "관점: 논문 심사"
2. 🔴 **치명적 문제 (Critical Issues)**: 버그, 리스크, 논리적 결함.
3. 🟡 **제안 사항 (Suggestions)**: 리팩토링, 최적화.
4. 🟢 **칭찬 포인트 (Praise)**: 긍정적인 피드백과 격려.
